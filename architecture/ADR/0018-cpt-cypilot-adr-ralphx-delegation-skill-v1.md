## Context and Problem Statement

Cypilot already knows how to generate agent integrations, compose skills from kits, and decompose large tasks into self-contained execution plans. However, actual execution of those plans still happens either directly in the current chat session or through host-tool-native subagents generated by Cypilot.

The user wants a dedicated integration with **ralphex** from `https://ralphex.com/`, not a different similarly named tool. ralphex is a standalone terminal CLI for autonomous plan execution with Claude Code. It runs from the root of a git repository, consumes Markdown plan files (normally under `docs/plans/`), executes tasks one by one in fresh Claude Code sessions, runs validation commands after each task, commits changes, and can continue with a multi-phase review pipeline, optional external Codex review, worktree isolation, and a web dashboard. It also supports optional Claude Code slash commands, but its primary integration surface is still the CLI plus local project files under `.ralphex/`.

The architectural question is: how should Cypilot integrate with the real ralphex product so that Cypilot remains the source of truth for SDLC guidance and planning, while ralphex becomes the autonomous execution backend for compiled plans?

## Decision Drivers

* **Explicit delegation path** — users need a first-class way to say "hand this task to ralphex" instead of having the current chat execute it directly
* **Preserve Cypilot as the source of truth** — SDLC rules, checklists, workflows, skills, and execution plans must remain canonical in Cypilot rather than being forked into a second rule system
* **Plan-to-execution continuity** — tasks decomposed by `cypilot-plan` must be transferable to ralphex's plan format with minimal manual rewriting
* **Low-friction setup** — the integration must support discovery, installation, and local project bootstrap for the real `ralphex` CLI
* **CLI-first compatibility** — the design must integrate through ralphex's documented CLI and file conventions (`docs/plans/`, `.ralphex/`) instead of assuming IDE-only or cloud-only control planes
* **Loose coupling** — base Cypilot workflows must remain usable without ralphex; ralphex must be an optional integration layer rather than a hard dependency of the core
* **Compact context handoff** — only the bounded slices needed for a delegated task should be compiled into the exported ralphex plan or optional `.ralphex/` overrides
* **Operational clarity** — ownership boundaries between planning, rule resolution, orchestration, execution, and validation must be explicit

## Considered Options

1. **Dedicated `cypilot-ralphex` skill with bounded delegation contract** — Cypilot keeps planning and rule resolution; a specialized skill discovers or installs ralphex, compiles Cypilot output into ralphex-compatible plans and optional `.ralphex/` overrides, then invokes ralphex explicitly
2. **Hard-wire ralphex into core generate/plan workflows** — the main Cypilot workflows invoke ralphex directly as part of ordinary execution, with no separate integration skill
3. **No integrated delegation path** — keep Cypilot execution chat-native and require users to manually copy plans, rules, and prompts into ralphex

## Decision Outcome

Chosen option: **Option 1 — Dedicated `cypilot-ralphex` skill with a bounded delegation contract**, because it adds the requested ralphex execution path while keeping Cypilot's planning and SDLC model authoritative, minimizing coupling to one external executor, and allowing projects to opt in only when they actually use ralphex.

The dedicated integration skill becomes the only ralphex-aware entry point inside Cypilot. Cypilot remains responsible for:

* identifying intent and target
* resolving kit paths and variables
* loading the correct SDLC rules, checklists, workflows, and skill slices
* producing execution plans and phase files
* defining deterministic validation commands and acceptance criteria
* compiling ralphex-compatible plan files and optional project-local `.ralphex/` overrides when customization is needed

ralphex becomes responsible for:

* autonomous task execution in fresh Claude Code sessions
* task retry loops and task-by-task validation execution
* git branch, commit, and plan-completion lifecycle handling
* multi-phase review orchestration, including optional external review
* worktree isolation and dashboard serving when enabled
* optional Claude slash commands as a convenience layer over the CLI

The delegation contract is **artifact-driven and CLI-native**, not a generic opaque packet or a mirror of the full Cypilot adapter. The `cypilot-ralphex` skill compiles Cypilot planning output into a ralphex-compatible Markdown plan and, when needed, a bounded set of `.ralphex/` prompt or agent overrides. The exported execution artifact contains only the minimum authoritative context ralphex needs, such as:

* a plan title and compact overview
* a `## Validation Commands` section derived from Cypilot's validation contract
* `### Task N:` sections with checkboxes derived from the relevant Cypilot phase work
* task-local guidance distilled from the matching Cypilot phase instructions
* resolved file paths and only the SDLC constraints needed for that delegated work
* optional generated files under `.ralphex/prompts/` or `.ralphex/agents/` when Cypilot needs to shape ralphex review behavior without forking the SDLC source of truth

### Consequences

* Good, because the user gets an explicit "delegate to ralphex" path instead of being forced into chat-native execution
* Good, because Cypilot remains the canonical owner of SDLC assets, execution plans, and validation contracts
* Good, because `cypilot-plan` outputs can be compiled into ralphex plan files rather than dying as dead-end planning artifacts
* Good, because the integration is additive — existing Cypilot generate/analyze/plan flows continue to work without ralphex installed
* Good, because bounded exported plans align with Cypilot's compact-prompts direction and avoid loading entire kits into ralphex unnecessarily
* Good, because installation and environment detection are concentrated in one integration surface instead of being scattered across core workflows
* Neutral, because the first high-value path is naturally centered on ralphex's documented CLI and Claude Code-oriented execution model
* Bad, because Cypilot must maintain a mapping layer between its own workflow semantics and ralphex's Markdown plan grammar and `.ralphex/` customization model
* Bad, because delegated execution introduces another external tool boundary that can fail independently (missing binary, broken install, invalid local config, incompatible ralphex version)
* Bad, because task status synchronization between ralphex runs and Cypilot plan state requires explicit contract design and tests

### Confirmation

Confirmed when:

- a dedicated ralphex-facing skill exists as a canonical Cypilot integration surface rather than embedding ralphex logic into general workflows
- the skill can discover an existing `ralphex` executable or guide installation via Homebrew, Go install, binary releases, or Docker-oriented instructions and persist the resolved executable path for reuse
- the skill can validate ralphex availability with a read-only diagnostics path before delegation
- the skill can bootstrap project-local `.ralphex/` configuration when needed, for example through `ralphex --init` or generated local overrides
- the skill can export a bounded delegated plan from canonical Cypilot sources into `docs/plans/` (or the configured `plans_dir`) without copying the entire SDLC kit into a second source tree
- a task planned through `cypilot-plan` can be delegated phase-by-phase or plan-by-plan to ralphex using a valid ralphex Markdown plan as the execution source of truth
- delegated runs can be validated afterward with the exact deterministic commands supplied by Cypilot for that task
- the integration remains optional: projects without ralphex continue to use normal Cypilot workflows with zero behavioral change

## Pros and Cons of the Options

### Option 1: Dedicated `cypilot-ralphex` skill with bounded delegation contract

Cypilot exposes ralphex through a specialized skill that handles tool discovery, installation, plan export, optional `.ralphex/` scaffolding, delegation, and post-run handoff, while core workflows remain executor-neutral.

* Good, because it creates a clear user-facing capability boundary: Cypilot plans and constrains, ralphex executes
* Good, because the ralphex integration can evolve independently without destabilizing core workflows
* Good, because SDLC assets remain canonical in Cypilot and are handed off by bounded slice or resolved reference instead of being duplicated wholesale
* Good, because `cypilot-plan` remains the authoritative decomposition engine and ralphex becomes an execution backend for exported plans
* Good, because installation and local bootstrap logic for ralphex live in one place
* Neutral, because it introduces a new integration skill that users must learn by name
* Bad, because the delegation packet format and status-sync behavior must be designed and tested explicitly

### Option 2: Hard-wire ralphex into core generate/plan workflows

Core workflows such as `generate.md` and `plan.md` invoke ralphex directly whenever execution crosses a threshold or a ralphex-capable environment is detected.

* Good, because there is no extra integration skill for the user to discover
* Good, because delegation could appear seamless when it works
* Bad, because it couples Cypilot's core workflow engine to one specific external orchestrator
* Bad, because projects that do not use ralphex would inherit extra branches, config, failure modes, and complexity in ordinary workflows
* Bad, because installation and environment management concerns would leak into general-purpose planning and generation logic
* Bad, because it becomes harder to support additional execution backends later

### Option 3: No integrated delegation path

Cypilot continues to plan and execute tasks in chat-native mode, and users manually move plans or prompts into ralphex when they want autonomous execution.

* Good, because Cypilot core remains simpler in the short term
* Good, because there is no ralphex-specific implementation work
* Bad, because it does not satisfy the requested user experience of assigning work specifically to ralphex rather than to the current chat
* Bad, because manual copying breaks determinism and increases the chance of stale or incomplete SDLC context
* Bad, because `cypilot-plan` outputs are not operationally integrated with an external executor
* Bad, because installation, plan-export, and `.ralphex/` setup remain ad hoc and error-prone

## More Information

### Architectural Boundaries

The integration follows these ownership rules:

| Concern | Owner |
|---|---|
| SDLC content (`rules.md`, `checklist.md`, templates, kit workflows, kit SKILL/AGENTS) | Cypilot |
| Intent routing and Protocol Guard | Cypilot |
| Execution plan generation and phase compilation | Cypilot |
| Delegation decision (`execute here` vs `delegate to ralphex`) | User + `cypilot-ralphex` skill |
| Autonomous task loop, retries, and reviews | ralphex |
| Dashboard, worktree mode, and long-running execution monitoring | ralphex |
| Deterministic post-run validation contract | Cypilot |

### ralphex Integration Surface

The integration should target ralphex's documented external surfaces rather than private implementation details:

* primary CLI execution: `ralphex <plan.md>`
* execution modes such as `--tasks-only`, `--review`, `--external-only`, `--worktree`, and `--serve`
* plan creation and local bootstrap commands such as `--plan` and `--init`
* documented file conventions: `docs/plans/`, `.ralphex/config`, `.ralphex/prompts/`, `.ralphex/agents/`
* optional Claude Code slash commands (`/ralphex`, `/ralphex-plan`, `/ralphex-update`) as convenience shims, not the primary contract

This keeps the integration aligned with ralphex's supported interfaces and avoids binding Cypilot to undocumented internals.

### Installation and Discovery Contract

The dedicated skill owns ralphex environment handling. The preferred order is:

1. reuse an existing `ralphex` executable already available on `PATH`
2. reuse a previously configured absolute executable path recorded by Cypilot
3. offer guided installation using the documented distribution paths — Homebrew on macOS, `go install`, binary releases, or Docker guidance when appropriate
4. record the resolved executable path so future delegation does not require re-discovery

Cypilot MUST NOT vendor ralphex into its own runtime or add ralphex as a mandatory dependency of the Cypilot Python package. ralphex remains an external optional tool.

### Plan-to-ralphex Mapping

`cypilot-plan` remains authoritative for decomposition. ralphex integration consumes plan outputs instead of replacing plan generation.

The mapping rule is:

* one Cypilot execution plan normally exports to one ralphex plan file under `docs/plans/` (or the configured `plans_dir`)
* one Cypilot phase normally maps to one `### Task N:` block or a small contiguous task group inside that exported plan
* Cypilot phase instructions, task steps, and acceptance criteria are flattened into ralphex-compatible checkboxes and validation commands
* `{cypilot_path}/.plans/{task}/out/` remains the stable interchange point for intermediate outputs that later export passes can consume

Cypilot therefore stays responsible for deciding **what the work decomposition is** and **what rules apply**, while ralphex is responsible for **executing the exported plan with fresh Claude sessions until done or stopped**.

The exported plan MUST satisfy ralphex's documented plan grammar: `## Validation Commands` for test/lint commands, `### Task N:` headers for executable work units, and checkboxes only inside task sections.

### SDLC Reuse Policy

ralphex must consume Cypilot SDLC guidance through **compiled plan text** and, when needed, **generated `.ralphex/` overrides derived from canonical Cypilot sources**, not by introducing a second independently maintained copy of the SDLC kit.

That means:

* no manual duplication of `rules.md`, `checklist.md`, kit workflows, or skill content into a parallel ralphex-maintained source tree
* no alternate normative rule set in ralphex that can drift from the Cypilot-managed SDLC kit
* exported plans may inline only the slices needed for the delegated task or phase
* if Cypilot needs custom ralphex review prompts or agents, it should generate them into project-local `.ralphex/` as derived artifacts rather than treating `.ralphex/` as the source of truth
* if a delegated task needs broader context, the packet must name the exact additional files or sections required rather than blindly exporting the whole adapter

### Applicability Notes

* **PERF**: Not applicable because this ADR selects an orchestration boundary and delegation model, not a performance-critical runtime algorithm
* **DATA**: Not applicable because this ADR does not choose a new persistent business-data model or migration strategy
* **COMPL**: Not applicable because this is an internal tool integration with no direct regulatory requirement introduced by the decision itself
* **SEC**: Applicable — the integration must avoid secrets in exported plans, must not hardcode risky Claude/Codex credentials or CLI flags into tracked artifacts, and must treat ralphex as an external executable boundary
* **INT**: Applicable — this ADR defines how Cypilot integrates with an external autonomous CLI and where the contract boundary lives
* **OPS**: Applicable — installation, environment diagnostics, and `.ralphex/` bootstrap behavior must be operationally clear and recoverable
* **MAINT**: Applicable — ownership is intentionally split so that Cypilot's SDLC model does not fork and the ralphex mapping layer stays localized
* **TEST**: Applicable — the exported plan contract, executable discovery, and Cypilot-plan-to-ralphex mapping must have automated regression coverage before acceptance
