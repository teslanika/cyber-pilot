## Context and Problem Statement

Cypilot already knows how to generate agent integrations, compose skills from kits, and decompose large tasks into self-contained execution plans. However, actual execution of those plans still happens either directly in the current chat session or through host-tool-native subagents generated by Cypilot.

The user now wants a different execution path: a dedicated Cypilot skill that can install and invoke RalphX, delegate work to RalphX instead of executing it inside the current chat, reuse Cypilot's SDLC assets (checklists, rules, workflows, skills), and run tasks that were previously decomposed by `cypilot-plan`. RalphX is an external CLI/MCP orchestrator for Ralph loops, with its own project registration, guardrails, import, monitoring, and loop execution model. The architectural question is: how should Cypilot integrate RalphX without coupling the whole core to a single external orchestrator or duplicating SDLC logic?

## Decision Drivers

* **Explicit delegation path** — users need a first-class way to say "hand this task to RalphX" instead of having the current chat execute it directly
* **Preserve Cypilot as the source of truth** — SDLC rules, checklists, workflows, skills, and execution plans must remain canonical in Cypilot rather than being forked into a second rule system
* **Plan-to-execution continuity** — tasks decomposed by `cypilot-plan` must be transferable to RalphX with minimal manual rewriting
* **Low-friction setup** — the integration must support discovery, installation, doctor-style validation, and project registration for RalphX
* **Loose coupling** — base Cypilot workflows must remain usable without RalphX; RalphX must be an optional integration layer rather than a hard dependency of the core
* **Compact context handoff** — only the bounded slices needed for a delegated task should be handed to RalphX, not entire kits or full Cypilot context
* **Operational clarity** — ownership boundaries between planning, rule resolution, orchestration, execution, and validation must be explicit

## Considered Options

1. **Dedicated `cypilot-ralphx` skill with bounded delegation contract** — Cypilot keeps planning and rule resolution; a specialized skill discovers or installs RalphX, exports only the task packet RalphX needs, registers the project, and delegates execution to RalphX
2. **Hard-wire RalphX into core generate/plan workflows** — the main Cypilot workflows invoke RalphX directly as part of ordinary execution, with no separate integration skill
3. **No integrated delegation path** — keep Cypilot execution chat-native and require users to manually copy plans, rules, and prompts into RalphX

## Decision Outcome

Chosen option: **Option 1 — Dedicated `cypilot-ralphx` skill with a bounded delegation contract**, because it adds the requested RalphX execution path while keeping Cypilot's planning and SDLC model authoritative, minimizing coupling to one external orchestrator, and allowing projects to opt in only when they actually use RalphX.

The dedicated integration skill becomes the only RalphX-aware entry point inside Cypilot. Cypilot remains responsible for:

* identifying intent and target
* resolving kit paths and variables
* loading the correct SDLC rules, checklists, workflows, and skill slices
* producing execution plans and phase files
* defining deterministic validation commands and acceptance criteria

RalphX becomes responsible for:

* long-running loop orchestration
* fresh-context iterative execution
* project registration inside RalphX
* step execution, pause/resume, and dashboard monitoring
* returning status and outputs back to the Cypilot-managed task context

The delegation contract is **bounded and artifact-driven**, not a full mirror of the Cypilot adapter. The `cypilot-ralphx` skill prepares a compact handoff packet for each delegated task or plan phase. That packet contains only the minimum authoritative context RalphX needs, such as:

* the target task or phase identity
* the resolved absolute file paths relevant to the task
* the exact Cypilot-derived rule/checklist/workflow/skill slices that govern the task
* the phase-local acceptance criteria
* the deterministic validation commands Cypilot expects after execution
* any prior-phase outputs from `{cypilot_path}/.plans/{task}/out/`

### Consequences

* Good, because the user gets an explicit "delegate to RalphX" path instead of being forced into chat-native execution
* Good, because Cypilot remains the canonical owner of SDLC assets, execution plans, and validation contracts
* Good, because `cypilot-plan` outputs become directly executable handoff units for RalphX rather than dead-end planning artifacts
* Good, because the integration is additive — existing Cypilot generate/analyze/plan flows continue to work without RalphX installed
* Good, because bounded task packets align with Cypilot's compact-prompts direction and avoid loading entire kits into RalphX unnecessarily
* Good, because installation and environment detection are concentrated in one integration surface instead of being scattered across core workflows
* Neutral, because the first high-value path is naturally centered on RalphX's documented CLI/MCP capabilities and Claude Code-oriented loop model
* Bad, because Cypilot must maintain a mapping layer between its own workflow semantics and RalphX's project/guardrail/import/run model
* Bad, because delegated execution introduces another external tool boundary that can fail independently (missing binary, broken env, invalid project registration, incompatible RalphX version)
* Bad, because task status synchronization between RalphX runs and Cypilot plan state requires explicit contract design and tests

### Confirmation

Confirmed when:

- a dedicated RalphX-facing skill exists as a canonical Cypilot integration surface rather than embedding RalphX logic into general workflows
- the skill can discover an existing `ralphx` executable or guide installation into an isolated Python environment and persist the resolved executable path for reuse
- the skill can validate RalphX availability with a read-only diagnostics path before delegation
- the skill can register the current project with RalphX and reuse that registration on subsequent runs
- the skill can export a bounded delegation packet from canonical Cypilot sources without copying the entire SDLC kit into a second source tree
- a task planned through `cypilot-plan` can be delegated phase-by-phase to RalphX using the plan manifest and phase files as the execution source of truth
- delegated runs can be validated afterward with the exact deterministic commands supplied by Cypilot for that task
- the integration remains optional: projects without RalphX continue to use normal Cypilot workflows with zero behavioral change

## Pros and Cons of the Options

### Option 1: Dedicated `cypilot-ralphx` skill with bounded delegation contract

Cypilot exposes RalphX through a specialized skill that handles tool discovery, installation, project registration, compact context export, delegation, and post-run handoff, while core workflows remain orchestrator-neutral.

* Good, because it creates a clear user-facing capability boundary: Cypilot plans and constrains, RalphX executes
* Good, because the RalphX integration can evolve independently without destabilizing core workflows
* Good, because SDLC assets remain canonical in Cypilot and are handed off by bounded slice or resolved reference instead of being duplicated wholesale
* Good, because `cypilot-plan` remains the authoritative decomposition engine and RalphX becomes an execution backend for those phases
* Good, because installation, version checks, and doctor logic for RalphX live in one place
* Neutral, because it introduces a new integration skill that users must learn by name
* Bad, because the delegation packet format and status-sync behavior must be designed and tested explicitly

### Option 2: Hard-wire RalphX into core generate/plan workflows

Core workflows such as `generate.md` and `plan.md` invoke RalphX directly whenever execution crosses a threshold or a RalphX-capable environment is detected.

* Good, because there is no extra integration skill for the user to discover
* Good, because delegation could appear seamless when it works
* Bad, because it couples Cypilot's core workflow engine to one specific external orchestrator
* Bad, because projects that do not use RalphX would inherit extra branches, config, failure modes, and complexity in ordinary workflows
* Bad, because installation and environment management concerns would leak into general-purpose planning and generation logic
* Bad, because it becomes harder to support additional execution backends later

### Option 3: No integrated delegation path

Cypilot continues to plan and execute tasks in chat-native mode, and users manually move plans or prompts into RalphX when they want RalphX orchestration.

* Good, because Cypilot core remains simpler in the short term
* Good, because there is no RalphX-specific implementation work
* Bad, because it does not satisfy the requested user experience of assigning work specifically to RalphX rather than to the current chat
* Bad, because manual copying breaks determinism and increases the chance of stale or incomplete SDLC context
* Bad, because `cypilot-plan` outputs are not operationally integrated with an external executor
* Bad, because installation, project registration, and guardrail setup remain ad hoc and error-prone

## More Information

### Architectural Boundaries

The integration follows these ownership rules:

| Concern | Owner |
|---|---|
| SDLC content (`rules.md`, `checklist.md`, templates, kit workflows, kit SKILL/AGENTS) | Cypilot |
| Intent routing and Protocol Guard | Cypilot |
| Execution plan generation and phase compilation | Cypilot |
| Delegation decision (`execute here` vs `delegate to RalphX`) | User + `cypilot-ralphx` skill |
| Long-running loop orchestration | RalphX |
| Dashboard, pause/resume, loop monitoring | RalphX |
| Deterministic post-run validation contract | Cypilot |

### RalphX Integration Surface

The integration should target RalphX's documented external surfaces rather than private implementation details:

* project registration and discovery
* diagnostics / doctor-style checks
* guardrail preview or validation flows
* markdown import flows for task packets or plan phases
* CLI or MCP-based loop execution
* status inspection and troubleshooting flows

This keeps the integration aligned with RalphX's supported interfaces and avoids binding Cypilot to RalphX internals.

### Installation and Discovery Contract

The dedicated skill owns RalphX environment handling. The preferred order is:

1. reuse an existing `ralphx` executable already available on `PATH`
2. reuse a previously configured absolute executable path recorded by Cypilot
3. offer guided installation into an isolated Python environment
4. record the resolved executable path so future delegation does not require re-discovery

Cypilot MUST NOT vendor RalphX into its own runtime or add RalphX as a mandatory dependency of the Cypilot package. RalphX remains an external optional tool.

### Plan-to-RalphX Mapping

`cypilot-plan` remains authoritative for decomposition. RalphX integration consumes plan outputs instead of replacing plan generation.

The mapping rule is:

* one Cypilot plan directory = one delegated RalphX project workflow context
* one Cypilot phase file = one RalphX-executable work unit or imported markdown task
* `{cypilot_path}/.plans/{task}/out/` = the stable interchange point for intermediate outputs that later phases consume

Cypilot therefore stays responsible for deciding **what the phases are** and **what rules apply**, while RalphX is responsible for **running the phase work with fresh contexts until done or stopped**.

### SDLC Reuse Policy

RalphX must consume Cypilot SDLC guidance through **bounded exported slices** or **resolved references prepared by Cypilot**, not by introducing a second independently maintained copy of the SDLC kit.

That means:

* no manual duplication of `rules.md`, `checklist.md`, kit workflows, or skill content into a parallel RalphX-maintained source tree
* no alternate normative rule set in RalphX that can drift from the Cypilot-managed SDLC kit
* task packets may inline only the slices needed for the delegated task or phase
* if a delegated task needs broader context, the packet must name the exact additional files or sections required rather than blindly exporting the whole adapter

### Applicability Notes

* **PERF**: Not applicable because this ADR selects an orchestration boundary and delegation model, not a performance-critical runtime algorithm
* **DATA**: Not applicable because this ADR does not choose a new persistent business-data model or migration strategy
* **COMPL**: Not applicable because this is an internal tool integration with no direct regulatory requirement introduced by the decision itself
* **SEC**: Applicable — the integration must avoid secrets in exported task packets, must not hardcode tokens, and must treat RalphX as an external executable boundary
* **INT**: Applicable — this ADR defines how Cypilot integrates with an external CLI/MCP orchestrator and where the contract boundary lives
* **OPS**: Applicable — installation, environment diagnostics, and project registration must be operationally clear and recoverable
* **MAINT**: Applicable — ownership is intentionally split so that Cypilot's SDLC model does not fork and the RalphX mapping layer stays localized
* **TEST**: Applicable — the delegation packet contract, executable discovery, and plan-phase mapping must have automated regression coverage before acceptance
